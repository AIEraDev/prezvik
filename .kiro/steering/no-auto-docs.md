# No Automatic Documentation Files

## Critical Rule

**DO NOT create markdown (.md) files to document your work unless explicitly requested by the user.**

This includes:

- Summary files
- Documentation files
- README updates
- Architecture docs
- Guide files
- Change logs
- Work reports

## Why?

Creating unsolicited documentation:

- Wastes time and resources
- Clutters the workspace
- Is annoying to clean up
- The user just wants the work done

## What to Do Instead

When you complete work:

1. Provide a brief verbal summary (2-3 sentences max)
2. Only mention key changes
3. Let the user ask for documentation if needed

## Exceptions

You MAY create .md files when:

- User explicitly asks for documentation
- User requests a README, guide, or summary file
- User says "document this" or similar
- It's part of the project structure (e.g., API docs)

## Examples

❌ **Bad**: "I've built the feature. Here's IMPLEMENTATION.md..." ✅ **Good**: "Built the feature with X, Y, Z. Ready to use."

❌ **Bad**: Creating ARCHITECTURE.md after building a system ✅ **Good**: Just build the system and briefly explain what was done

Remember: Users want results, not paperwork.
